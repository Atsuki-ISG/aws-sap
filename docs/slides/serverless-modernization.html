試験では「最小限の変更で」→ Rehost/Replatform、「スケーラビリティ向上」「運用負荷削減」→ Refactor (サーバーレス化) が問われる。

ひっかけパターン & 練習問題

よくあるひっかけ 5選

1. 「全てを Lambda に移行」
15分超の処理、常時高負荷のワークロードは Lambda に不向き。Fargate / ECS を検討。
2. 「サーバーレス = DynamoDB に移行」
複雑なリレーショナルクエリやACIDトランザクション → Aurora Serverless v2 が正解。
3. 「Lambda + RDS を直接接続」
同時実行数に応じてDB接続が爆発 → RDS Proxy が必須。これがないと不正解。
4. 「Step Functions Express で長時間処理」
Express は最大 5 分。長時間ワークフロー → Standard を選択。
5. 「一括でマイクロサービス化」
ビッグバン移行はリスク大。Strangler Fig で段階的に切り出すのが正解パターン。

練習問題 1

EC2上のモノリスアプリをサーバーレス化したい。RDB (PostgreSQL) で複雑なJOINクエリを多用。一部APIは応答時間100ms以内のSLAがある。データ層とコンピュート層の最適な構成は？
正解: C
A: DynamoDB は複雑な JOIN に不向き。GSI でカバーできるのは限定的。
B: Lambda のコード内接続プールは実行コンテキスト単位で効果が限定的。RDS Proxy がないと接続数爆発のリスク。
C: Aurora Serverless v2 で RDB 要件を満たし、RDS Proxy で接続管理。Provisioned Concurrency で 100ms SLA 対応。
D: Fargate は常時起動でコスト増。サーバーレス化の要件に合わない。API の SLA 対応には Lambda + Provisioned Concurrency が最適。

練習問題 2

複数のマイクロサービスで構成されたECサイト。注文確定後に (1) 在庫更新 (2) 決済処理 (3) メール通知 (4) 配送手配 を順次実行する必要がある。決済失敗時は在庫をロールバックする。最適な構成は？
正解: C
A: SNS はファンアウト (並列通知)。順次実行やロールバックに不向き。
B: SQS チェーンは順次処理可能だが、エラー時のロールバック実装が複雑。可視性も低い。
C: Step Functions で Saga パターン。各ステップに補償アクション (在庫ロールバック等) を定義。実行状況の可視化とエラーハンドリングが容易。
D: EventBridge はコレオグラフィ型で疎結合。順次実行の保証やロールバック制御には不向き。
