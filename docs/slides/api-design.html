ひっかけ注意
「HTTP API」は名前から機能が豊富に見えるが、REST APIの方が機能が多い。
HTTP APIは「REST APIの軽量版」。WAF・キャッシュ・使用量プランが不要ならHTTP APIでコスト削減。

Resolver タイプ

Unit Resolver

単一のデータソースに対する操作。DynamoDBのGetItem、PutItemなどを直接マッピング。

Pipeline Resolver

複数のデータソースを順次呼び出し。認可チェック後にデータ取得、などの多段処理に対応。

リアルタイム: Subscription

Subscription の仕組み
クライアントがSubscriptionを登録 → Mutation発生時にWebSocket経由で自動プッシュ。
API Gateway WebSocketとの違い: AppSyncはGraphQLスキーマベースで宣言的に定義。接続管理の自前実装が不要。

API Gateway の応用パターン

Lambda統合だけでなく、サービス直接統合やカスタムドメインなど試験で問われる応用を押さえる。

統合タイプ

Lambda プロキシ統合

最も一般的。リクエスト全体をLambdaに渡す。レスポンス形式はLambda側で制御。

AWSサービス直接統合

LambdaなしでDynamoDB / SQS / SNS / Step Functions / Kinesisを直接呼び出し。
コスト削減+低レイテンシ。

HTTP統合

バックエンドのHTTPエンドポイント(ALB等)にプロキシ。VPC Link経由でプライベートリソースへ。

Mock統合

バックエンド不要で固定レスポンス。開発・テスト用。CORS対応のOPTIONSにも利用。

ひっかけパターン & 練習問題

よくあるひっかけ 5選

1. 「HTTP APIがREST APIより高機能」
名前に騙されない。HTTP APIはREST APIの軽量・低コスト版。WAF・キャッシュ・使用量プランはREST APIのみ。
2. 「API GatewayでGraphQLを提供」
API GatewayはREST/HTTPのみ。GraphQLが要件 → AppSync。API GatewayのPOSTエンドポイントでGraphQLをプロキシはアンチパターン。
3. 「AppSyncで外部API公開+課金管理」
AppSyncには使用量プランがない。外部API公開+利用量制御 → API Gateway REST API。
4. 「WebSocket APIでオフライン対応」
WebSocket APIは常時接続が前提。オフライン対応 → AppSync + Amplify DataStore。
5. 「HTTP APIでプライベートAPI」
プライベートAPI (VPC Endpoint経由) はREST APIのみ。HTTP APIは非対応。

練習問題 1

モバイルアプリ開発チームが、複数のDynamoDBテーブルとRDS Aurora から1回のAPIコールでデータを取得したい。オフライン時もアプリ操作を可能にし、オンライン復帰時に自動同期する。最適な構成は？
正解: B
A: 複数データソース統合は可能だが、オフライン対応がない。クエリの柔軟性もREST固定。
B: AppSyncのPipeline ResolverでDynamoDB+Auroraを宣言的に統合。Amplify DataStoreでオフライン+自動同期。全要件を満たす。
C: WebSocket APIはリアルタイム通信用。オフライン対応や複数データソース統合には不向き。
D: API GatewayでGraphQLをプロキシするのはアンチパターン。AppSyncのネイティブ機能を使うべき。

練習問題 2

SaaS企業が外部パートナーにREST APIを公開する。要件: (1) パートナーごとにリクエスト上限を設定 (2) WAFでAPIを保護 (3) レスポンスキャッシュでバックエンド負荷を軽減 (4) 最小コスト。最適な構成は？
正解: B
A: HTTP APIには使用量プランがなく、パートナーごとのリクエスト上限設定ができない。WAFもHTTP APIに直接適用不可。
B: REST APIなら使用量プラン(パートナーごとのAPIキー+上限)、WAF統合、レスポンスキャッシュの全てをネイティブで対応。
C: AppSyncはGraphQL用。RESTful API公開と使用量プラン(パートナー課金管理)には不向き。
D: ALBベースだと使用量プラン相当の機能を自前実装する必要があり、運用負荷が高い。
