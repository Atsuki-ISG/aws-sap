PrivateLink の特徴

NLBベース

プロバイダー側にNLB (Network Load Balancer) が必須。GWLBも可。ALBは不可。

IP重複環境で活用

Peering/TGWと異なりCIDRが重複していても接続可能。ENIのプライベートIPで通信。

一方向アクセス

コンシューマー → プロバイダーへの一方向のみ。双方向通信には双方でEndpoint Serviceを構築。

クロスアカウント対応

別アカウントへのサービス公開が可能。許可リストでアクセス元を制御。

PrivateLink vs VPC Peering / TGW 判断
フルルーティング不要で特定サービスだけ公開 → PrivateLink
IP重複がある環境 → Peering/TGW不可 → PrivateLink
全VPC間の通信が必要 → TGW（PrivateLinkはサービス単位）
ひっかけ注意
PrivateLinkのプロバイダー側はALBではなくNLB。「ALBの背後にあるアプリをPrivateLinkで公開」→ NLBを前段に追加するか、ALBをNLBのターゲットにする。

Resolver Rule の共有
Resolver RuleはAWS RAMで他アカウントと共有可能。
マルチアカウント環境では共有サービスアカウントにResolverを配置し、ルールを組織全体に共有するのがベストプラクティス。
覚え方
Inbound = AWSに入ってくるDNS問い合わせ（オンプレ→AWS）
Outbound = AWSから出ていくDNS問い合わせ（AWS→オンプレ）
方向はDNSクエリの向きで判断。データの流れではない。

Split-View DNS

Split-View DNS（同一ドメイン名で内外を分ける）
同じドメイン名（例: api.example.com）に対して：
Public Hosted Zone：インターネット向け → パブリックIP/CloudFront
Private Hosted Zone：VPC内向け → プライベートIP/内部ALB
VPC内からの問い合わせはPHZが優先される（PHZとPHZが同名の場合）。

ひっかけパターン & 練習問題

よくあるひっかけ 5選

1. 「Gateway EndpointでオンプレからS3アクセス」
Gateway EPはVPC内ルートテーブルベース。オンプレからは到達不可。→ Interface Endpoint for S3。
2. 「ALBでPrivateLinkを構築」
PrivateLinkのプロバイダー側はNLBまたはGWLBが必須。ALBは直接使えない。
3. 「TGWでIP重複VPCを接続」
TGWはルーティングベース。CIDR重複は不可。→ PrivateLinkならIP重複でもOK。
4. 「コンソールからPHZをクロスアカウント共有」
CLI/SDK必須。マネジメントコンソールではクロスアカウント関連付けは不可。
5. 「Outbound EPでオンプレからAWSの名前解決」
方向が逆。オンプレ→AWSはInbound。Outboundは AWS→オンプレ。

練習問題 1

企業がオンプレDCからDirect Connect経由でS3バケットにプライベートアクセスしたい。インターネットを経由せず、コスト効率の良い構成は？
正解: B
A: Gateway EPはVPC内のルートテーブルベース。オンプレからの通信はルートテーブルを経由しない。
B: Interface EP（PrivateLink for S3）はENIにプライベートIPを持つため、DX/VPN経由でオンプレから到達可能。
C: Public VIFは技術的に可能だが「インターネットを経由せず」の要件でプライベートアクセスが求められている。
D: S3はAWSマネージドサービス。自前でEndpoint Serviceを構築する必要はない。

練習問題 2

マルチアカウント環境（Organizations）で、全アカウントのVPCからオンプレDNSサーバー（corp.local）の名前解決を可能にしたい。最も運用負荷が低い構成は？
正解: C
A: 各アカウントに個別構築は運用負荷が高い。スケールしない。
B: DHCPオプションでオンプレDNSを指定するとAWS内部DNSが使えなくなる。Private HZ等が解決できなくなる。
C: Resolver Ruleは RAM で共有可能。1箇所でOutbound EP + ルールを管理し、全アカウントに配布。最も運用負荷が低い。
D: DNSトラフィックのルーティングだけではResolver統合にならない。構成が複雑で運用負荷が高い。
