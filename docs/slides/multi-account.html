クロスアカウントアクセスの2パターン

リソースポリシー方式

S3バケットに他アカウントのPrincipalを指定 → 相手は自分の権限のままアクセス可能。ロール切替不要。

AssumeRole 方式

対象アカウントにIAMロールを作成 → 信頼ポリシーで許可 → 呼び出し元がsts:AssumeRole → 一時認証情報で操作。

重要: リソースポリシーの利点
リソースポリシー方式では、呼び出し元が元の権限を保持したままアクセスする。AssumeRoleだと元の権限を「放棄」してロールの権限に切り替わる。
KMSのクロスアカウント復号はキーポリシー(リソースポリシー) + IAMポリシーの両方が必要。
ひっかけ注意
「DynamoDBに他アカウントからアクセス」→ DynamoDBにはリソースポリシーがない → AssumeRole一択。
リソースポリシーがないサービスでは、必ずIAMロール経由のクロスアカウントになる。

SCP設計の鉄則
1. DenyList戦略: FullAWSAccessを残し、禁止事項だけDenyで追加 (推奨)
2. AllowList戦略: FullAWSAccessを外し、許可するサービスのみ列挙 (厳格だが運用コスト大)
試験ではDenyList戦略が「最も運用負荷が低い」選択肢として頻出。
ひっかけ注意
SCPの継承は「AND条件（交差）」。親OUがDenyしたものは子OUでAllowしても拒否される。
管理アカウントにはSCPが適用されないため、管理アカウントでのワークロード実行は非推奨。

Control Tower の役割と制約

Control Tower は Organizations 上にベストプラクティスのランディングゾーンを自動構築するサービス。

Control Tower が提供するもの

Landing Zone

マルチアカウント環境の初期セットアップ。Security OU + Log Archive / Audit アカウントを自動作成。

予防的ガードレール

SCPで実装。操作を事前に禁止。
例: リージョン制限、CloudTrail無効化禁止。

検出的ガードレール

AWS Configルールで実装。違反を検知して通知。
例: S3パブリックアクセス検知、MFA未有効化。

Control Tower vs 手動 Organizations の判断
Control Tower を選ぶ: ベストプラクティスに沿った構築 / ガードレール一括管理 / Account Factory でアカウント量産

「Control Tower の予防的ガードレール」=「SCP」であることを理解していないと誤答する。

ひっかけパターン & 練習問題

よくあるひっかけ 5選

1. 「SCPで権限を許可する」
SCPは許可を付与しない。許可の「上限」を設定するだけ。実際の許可はIAMポリシーが必要。
2. 「管理アカウントをSCPで制限」
管理アカウントにはSCPが適用されない。管理アカウントでワークロードを動かすべきではない。
3. 「検出的ガードレールで操作をブロック」
検出的 = AWS Config = 通知のみ。ブロックは予防的(SCP)で行う。
4. 「DynamoDBにリソースポリシーを設定」
DynamoDB, EC2, RDS等にはリソースポリシーがない。クロスアカウントはAssumeRole。
5. 「SCPでIAMユーザーの権限を昇格」
SCPは制限のみ。子OUで親の制限を解除することはできない。

練習問題 1

ある企業は AWS Organizations を使用し、開発チームに各自のアカウントで IAM ロールを自由に作成させたい。ただし、作成されるロールが AdministratorAccess を持つことは防ぎたい。最も適切な方法は？
正解: C
A: CreateRoleを完全制限すると、チームがロールを作れなくなる。
B: 検知・削除は事後対応。作成自体は防げず、一時的にAdmin権限が存在する。
C: Permission Boundaryで上限を設定し、SCPでPB付与を強制。IAM委譲+制限の最適解。
D: 検出的ガードレールは通知のみで禁止はできない。

練習問題 2

アカウントAのLambda関数が、アカウントBのS3バケットからオブジェクトを読み取る必要がある。Organizations内の同一組織。最も運用負荷が低い方法は？
正解: B
A: 技術的に正しいが、AssumeRoleの実装が必要で運用負荷が高い。
B: S3バケットポリシー(リソースポリシー)でPrincipal指定のみ。Lambdaのコード変更不要。最もシンプル。
C: S3はRAMの共有対象外。RAMはVPCサブネット、TGW、Route 53等が対象。
D: SCPは許可を付与しない。アクセス権限の付与にSCPは使えない。
